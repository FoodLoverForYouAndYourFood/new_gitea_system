# Специалист: модели процессов и потоков

Упрощённые нотации, которые работают в команде без перегрузки.

## Когда BPMN
- Нужно показать шаги процесса, роли и ветки.
- Важны события/таймеры/ошибки (например, платёжные процессы).

Мини-легенда BPMN:
- Прямоугольник — задача, ромб — развилка, круг — событие.
- Lane/Pool — роли/системы.
- Сообщение — внешняя система, таймер — задержка.

## Когда UML Activity/Sequence
- Activity — показать поток внутри системы без бизнес-ролей.
- Sequence — показать обмены между системами/сервисами по времени.

Мини-легенда Sequence:
- Lifeline — участник (сервис/пользователь).
- Сообщение — запрос/ответ, указывать протокол/событие.
- Alt/Opt блоки — ветвления.

## Хорошая схема
- 8–20 блоков, не больше двух экранов.
- Подписи простые: глагол + объект («Проверить лимит», «Создать заказ»).
- Цвет — только для выделения рисков/ручных шагов.

## Быстрый процесс AS-IS/TO-BE
- AS-IS: фиксируем реальность, боли и задержки.
- TO-BE: целевое, убираем лишние шаги, добавляем автоматизацию.
- Разница между AS-IS и TO-BE — аргумент для приоритизации.

## Документация к схеме
- Цель схемы (что отвечаем).
- Описание ролей/систем.
- Неочевидные правила/валидации.
- Открытые вопросы по шагам (выделить цветом).

## Чек-лист схемы
- [ ] Понятно, кто инициирует процесс.
- [ ] Видны внешние системы и ручные шаги.
- [ ] Решения/ветки подписаны по условиям.
- [ ] Есть пометки по ошибкам/ретраям, если важно.
- [ ] AS-IS и TO-BE отличимы.
